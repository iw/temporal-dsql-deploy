# Requirements Document

## Introduction

Rewrite the `temporal-dsql-deploy` CLI from Python (Typer/uv) to Rust, producing a single binary `dsqld` that manages the full local development lifecycle: infrastructure provisioning, image building, schema setup, and Docker Compose orchestration. The rewrite converges on patterns established in `temporal-dsql-deploy-eks` (AWS SDK for infra, TOML config, Dagger client) and `temporal-loom` (workspace CLI, exec module, compose wrapping). Terraform is replaced by direct AWS SDK calls. Hand-edited `.env` files are replaced by a `config.toml` that serves as the single source of truth, with `.env` generated as a derived artifact.

## Glossary

- **CLI**: The `dsqld` command-line binary built from the `crates/cli/` crate
- **Config_Model**: The TOML configuration model in `crates/config/`, deserialized via serde into Rust structs
- **Config_File**: The `config.toml` file at the workspace root, the single source of truth for all deployment parameters
- **Env_File**: The `dev/.env` file, a derived artifact generated by the CLI from Config_File before running Docker Compose
- **Infra_Module**: The infrastructure provisioning code in `crates/cli/` that uses AWS SDK directly (replacing Terraform)
- **Build_Binary**: The `dsqld-build` binary built from `crates/build/`, a separate Dagger-based image builder
- **Dagger_Client**: The lightweight GraphQL client in `crates/dagger-client/`, copied from `temporal-dsql-deploy-eks`
- **Workspace_Root**: The compile-time constant injected via `.cargo/config.toml` pointing to the repository root
- **Dev_Directory**: The `dev/` directory containing `docker-compose.yml`, generated `.env`, config files, and dynamic config
- **DSQL_Cluster**: An Aurora DSQL cluster provisioned via the AWS SDK
- **DynamoDB_Tables**: Optional DynamoDB tables for distributed rate limiting and connection leasing
- **Schema_Tool**: The `temporal-dsql-tool` binary used to apply DSQL schema
- **Compose_Stack**: The Docker Compose service stack defined in `dev/docker-compose.yml`

## Requirements

### Requirement 1: Cargo Workspace Structure

**User Story:** As a developer, I want the project organized as a Cargo workspace with well-defined crates, so that each concern is isolated and the codebase follows the same patterns as sibling repos.

#### Acceptance Criteria

1. THE CLI SHALL be organized as a Cargo workspace with four member crates: `cli`, `config`, `build`, and `dagger-client`
2. THE CLI crate SHALL produce a binary named `dsqld`
3. THE Build_Binary crate SHALL produce a binary named `dsqld-build`
4. THE Dagger_Client crate SHALL be copied from `temporal-dsql-deploy-eks/crates/dagger-client/` with no functional changes
5. THE workspace SHALL use Rust edition 2024 and the stable toolchain
6. THE workspace SHALL pass `cargo fmt` and `cargo clippy -- -D warnings` with zero warnings
7. THE workspace SHALL define a `.cargo/config.toml` that sets a `DSQLD_WORKSPACE_ROOT` environment variable at compile time
8. THE CLI crate SHALL use `clap` with derive macros for argument parsing, `eyre` and `color-eyre` for error handling, and `which` for tool detection
9. THE Config_Model crate SHALL use `serde` and `toml` for deserialization, and `thiserror` for error types
10. THE Build_Binary crate SHALL use `clap` for argument parsing, `eyre` for error handling, and Dagger_Client for image builds

### Requirement 2: TOML Configuration as Single Source of Truth

**User Story:** As a developer, I want a single `config.toml` file that drives all CLI commands, so that I no longer need to hand-edit `.env` files or keep multiple configuration sources in sync.

#### Acceptance Criteria

1. THE Config_Model SHALL deserialize a `config.toml` file into typed Rust structs with serde
2. THE Config_File SHALL contain sections for: `[project]` (name, region), `[dsql]` (endpoint, connection pool settings, reservoir settings), `[elasticsearch]` (host, port, index), `[temporal]` (log level, history shards, image name), and `[dynamodb]` (optional rate limiter and connection lease table names)
3. THE Config_Model SHALL provide sensible defaults for all fields except `dsql.endpoint` (which is populated by `infra apply`)
4. WHEN `dsqld config init` is run, THE CLI SHALL generate a `config.toml` file with documented defaults and placeholder values
5. WHEN `dsqld config init` is run AND a `config.toml` already exists, THE CLI SHALL refuse to overwrite the existing file and report an error
6. THE CLI SHALL provide a `config.example.toml` at the workspace root as a reference for all configuration options
7. FOR ALL valid Config_File instances, serializing to TOML then deserializing SHALL produce an equivalent Config_Model (round-trip property)
8. THE Config_Model SHALL validate that `dsql.max_idle_conns` equals `dsql.max_conns` and report a clear error when the invariant is violated

### Requirement 3: Environment File Generation

**User Story:** As a developer, I want the CLI to generate the `.env` file from `config.toml` automatically, so that Docker Compose always uses configuration consistent with my TOML settings.

#### Acceptance Criteria

1. WHEN any `dsqld dev` subcommand is run, THE CLI SHALL generate `dev/.env` from the current Config_File before invoking Docker Compose
2. THE CLI SHALL map Config_Model fields to the environment variable names expected by the Docker Compose stack (matching the format in the current `.env.example`)
3. THE Env_File SHALL include all variables required by the Compose_Stack: DSQL connection settings, Elasticsearch settings, AWS region, Temporal settings, reservoir settings, and Docker image name
4. WHEN the Config_File is missing required fields (such as `dsql.endpoint`), THE CLI SHALL report a clear error identifying the missing field and refuse to generate the Env_File
5. FOR ALL valid Config_Model instances, generating an Env_File then parsing the key-value pairs back SHALL produce values matching the original Config_Model fields (round-trip property)

### Requirement 4: Infrastructure Provisioning via AWS SDK

**User Story:** As a developer, I want to provision DSQL clusters and optional DynamoDB tables directly via the AWS SDK, so that I no longer depend on Terraform for infrastructure management.

#### Acceptance Criteria

1. WHEN `dsqld infra apply` is run, THE Infra_Module SHALL create a DSQL_Cluster using `aws-sdk-dsql` with deletion protection enabled and appropriate tags
2. WHEN `dsqld infra apply` is run, THE Infra_Module SHALL create two DynamoDB_Tables (rate limiter and connection lease) using `aws-sdk-dynamodb` with on-demand billing, TTL enabled, and appropriate tags â€” DynamoDB tables are always provisioned because this repo exercises the full DSQL connection management stack
3. WHEN `dsqld infra apply` completes successfully, THE CLI SHALL write the DSQL endpoint back into Config_File under `dsql.endpoint`
4. WHEN `dsqld infra destroy` is run, THE CLI SHALL require explicit confirmation before deleting any resources
5. WHEN `dsqld infra status` is run, THE CLI SHALL display the current state of provisioned resources (DSQL cluster endpoint, DynamoDB table names if present) by querying the AWS APIs
6. THE Infra_Module SHALL use the project name from Config_File as a prefix for all AWS resource names and tags
7. THE Infra_Module SHALL use `tokio` for async AWS SDK operations
8. IF an AWS API call fails with a retryable error, THEN THE Infra_Module SHALL retry with the SDK's built-in retry policy
9. IF an AWS API call fails with a permission error, THEN THE Infra_Module SHALL report a clear error message with remediation hints

### Requirement 5: Dagger-Based Image Building

**User Story:** As a developer, I want to build Temporal DSQL Docker images via Dagger using the same patterns as the EKS and Loom repos, so that image builds are consistent across all deployment targets.

#### Acceptance Criteria

1. WHEN `dsqld build temporal` is run, THE CLI SHALL delegate to the Build_Binary (`dsqld-build`) as a subprocess
2. THE Build_Binary SHALL locate the `temporal-dsql` source repository via `--source` argument, `TEMPORAL_DSQL_PATH` environment variable, or default to `../temporal-dsql`
3. THE Build_Binary SHALL validate that the source directory contains a `go.mod` file and report a clear error if the directory does not appear to be a Go repository
4. THE Build_Binary SHALL extract the Go version from `go.mod` (major.minor only) and use it to select the builder image
5. THE Build_Binary SHALL compile `temporal-server` and `temporal-dsql-tool` Go binaries with CGO disabled and the `disable_grpc_modules` build tag
6. THE Build_Binary SHALL produce two Docker images: `temporal-dsql-server:latest` (base image with binaries and runtime config) and `temporal-dsql-tool:latest` (minimal image with the tool binary only)
7. THE Build_Binary SHALL accept an `--arch` argument defaulting to `arm64` for target architecture selection
8. FOR ALL valid `go X.Y.Z` directives in a `go.mod` file, the Go version extraction SHALL produce the string `X.Y` (property: major.minor extraction)

### Requirement 6: Schema Setup

**User Story:** As a developer, I want to run DSQL schema setup through the CLI, so that I have a single tool for the entire development workflow.

#### Acceptance Criteria

1. WHEN `dsqld schema setup` is run, THE CLI SHALL execute the Schema_Tool binary as a subprocess with connection parameters derived from Config_File
2. THE CLI SHALL pass the DSQL endpoint, port, user, database, and AWS region from Config_File to the Schema_Tool
3. THE CLI SHALL accept `--version` (default `1.1`), `--overwrite`, and `--tool` (path to Schema_Tool binary) options
4. IF the Schema_Tool binary is not found at the specified path, THEN THE CLI SHALL report a clear error with a hint to build it first
5. IF the Schema_Tool process exits with a non-zero code, THEN THE CLI SHALL propagate the exit code and report the failure

### Requirement 7: Docker Compose Lifecycle Management

**User Story:** As a developer, I want to manage the Docker Compose dev stack through the CLI with commands that match the patterns in `temporal-loom`, so that the workflow is familiar across repos.

#### Acceptance Criteria

1. THE CLI SHALL provide `dsqld dev up`, `dsqld dev down`, `dsqld dev ps`, `dsqld dev logs`, and `dsqld dev restart` subcommands
2. WHEN `dsqld dev up` is run, THE CLI SHALL generate the Env_File from Config_File, then invoke `docker compose -f dev/docker-compose.yml up`
3. THE `dsqld dev up` command SHALL accept a `--detach` (`-d`) flag to run services in the background
4. THE `dsqld dev down` command SHALL accept a `--volumes` (`-v`) flag to remove volumes
5. THE `dsqld dev logs` command SHALL accept an optional service name argument and a `--follow` (`-f`) flag
6. THE `dsqld dev restart` command SHALL accept optional service name arguments to restart specific services
7. THE CLI SHALL verify that `docker` is available on PATH before attempting any compose operation and report a clear error if missing
8. THE CLI SHALL run all compose commands from the Workspace_Root directory
9. IF a compose command exits with a non-zero code, THEN THE CLI SHALL propagate the exit code

### Requirement 8: Directory Structure Migration

**User Story:** As a developer, I want the profile concept flattened into a single `dev/` directory, so that the project structure is simpler and matches the single-profile reality.

#### Acceptance Criteria

1. THE project SHALL use a single `dev/` directory containing `docker-compose.yml`, generated `.env`, `config/` (Alloy, Mimir, Grafana configs), and `dynamicconfig/`
2. THE `dev/docker-compose.yml` SHALL reference config files relative to `dev/` and Docker assets relative to the workspace root (e.g., `../docker/`)
3. THE `dev/docker-compose.yml` SHALL NOT hardcode any DSQL, reservoir, rate limiting, or connection lease environment variables in the `environment:` block â€” all such configuration SHALL flow through the generated Env_File via `env_file:`
4. THE `dev/docker-compose.yml` SHALL use the Elasticsearch image `docker.elastic.co/elasticsearch/elasticsearch:8.17.0`, matching `temporal-loom`
5. THE CLI SHALL resolve all paths relative to Workspace_Root using a compile-time constant from `.cargo/config.toml`
6. THE `profiles/` directory SHALL be removed after migration to `dev/`

### Requirement 9: Python Artifact Removal

**User Story:** As a developer, I want all Python CLI artifacts removed after the Rust rewrite is complete, so that the repository has a single, clear toolchain.

#### Acceptance Criteria

1. WHEN the Rust CLI is feature-complete, THE project SHALL remove `src/tdeploy/`, the root `pyproject.toml`, `uv.lock`, `.python-version`, and any Python virtual environment artifacts
2. THE project SHALL retain `dsql-tests/` as an independent Python test suite
3. THE project SHALL add a `dsql-tests/pyproject.toml` with the Python dependencies previously declared in the root `pyproject.toml` that `dsql-tests/` requires (e.g., `temporalio`, `boto3`)
4. THE project SHALL update `AGENTS.md` and `README.md` to reflect the new Rust CLI commands and project structure

### Requirement 10: Terraform Removal

**User Story:** As a developer, I want Terraform removed entirely since infrastructure is now managed by the Rust CLI via AWS SDK.

#### Acceptance Criteria

1. THE project SHALL remove the `terraform/` directory and all its contents
2. THE project SHALL remove all references to Terraform from `AGENTS.md`, `README.md`, and any other documentation
3. THE project SHALL remove any Terraform-related helper code (e.g., `src/tdeploy/terraform.py`) as part of the Python artifact removal

### Requirement 11: Image Naming Convention

**User Story:** As a developer, I want Docker image names consistent with the EKS deployment repo, so that images are interchangeable across deployment targets.

#### Acceptance Criteria

1. THE Build_Binary SHALL produce images named `temporal-dsql-server:latest` and `temporal-dsql-tool:latest`
2. THE Config_Model SHALL default the Temporal image name to `temporal-dsql-server:latest`
3. THE `dev/docker-compose.yml` SHALL reference the new image names

### Requirement 12: Exec and Path Modules

**User Story:** As a developer, I want reusable modules for subprocess execution and path resolution matching the patterns in `temporal-loom`, so that the CLI has consistent, well-tested infrastructure code.

#### Acceptance Criteria

1. THE CLI crate SHALL include an `exec` module that runs subprocesses from Workspace_Root, streams stdio to the terminal, checks exit codes, and returns structured errors
2. THE CLI crate SHALL include a `paths` module that provides functions for resolving workspace-relative paths: `root()`, `compose_file()`, `env_file()`, `config_file()`, and `docker_dir()`
3. THE `exec` module SHALL use the `which` crate to verify tool availability before execution and report a clear error if a tool is missing
4. THE `exec` module SHALL print the command being executed (prefixed with `â–¸`) to stderr before running the subprocess

### Requirement 13: Rust Code Quality Standards

**User Story:** As a developer, I want the Rust codebase to follow the same quality standards as sibling repos, so that code is consistent, safe, and maintainable.

#### Acceptance Criteria

1. THE workspace SHALL use `thiserror` for error types in the Config_Model crate and `eyre` for error handling in CLI and Build_Binary crates
2. THE workspace SHALL contain no `.unwrap()` calls outside of test code
3. THE workspace SHALL derive `Debug` on all public types
4. THE workspace SHALL use `tracing` for structured logging in library code and `eprintln!` for user-facing output in CLI crates
5. THE workspace SHALL contain no `unsafe` code
